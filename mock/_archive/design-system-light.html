toritavi · Design Direction

Daylight — 明るく、軽く、大人。

白い余白と細い罫で「軽さ」を、引き上げた teal で「明るさ」を、抑えた彩度・semibold 見出し・点だけの暖色で「大人っぽさ」を出す方向。現行 DS の構成（カラー → タイポ → Radius/Shadow → アイコン → コンポーネント → プレビュー）を踏襲したサンプル。

余白で軽さ（影に頼らない） 彩度は中庸・ネオン化しない 暖色は"点"だけ 見出しは semibold（800 を捨てる） 角丸 16・bubbly にしない ヘアライン罫 + 極薄シャドウ

12. 適用ガイドライン

Daylight トークンを実装に落とすときの判断基準。teal は CTA・active・確定に "点" で、明るさは白と余白で。色面でなくアイコン形で種別を識別する。

12.1 DO / DON'T
DO
・アイコンは inline SVG（viewBox 0 0 24 24 / stroke 1.8 / round）で描画
・サムネは --t-a14+teal-700 icon（主役）か --n-100+body icon の2トーン。種別はアイコン形で識別
・角丸は基本 --r-md (16px)、面の持ち上げは --sh-card かヘアライン罫で
・価格・CTA・リンクは teal を点で（CTA面=--t-500、文字=--t-700）
・入力欄は左にラベル、focus は --t-ring リング
・タイムラインはドット+縦線、完了=teal、進行中=teal active
・状態は意味色トークンで: --suc-* / --wrn-* / --dng-* / teal
DON'T
・emoji アイコンを UI に残さない（サムネ含む）
・複数の角丸値を無秩序に混在させない（sm/md/lg/pill の段以外）
・teal をカード・背景にベタ塗りしない（点でアクセント）
・旧 ink ネイビーの dark/gradient ヒーローを使わない（teal-deep グラデに置換）
・カードに強い影を重ねない／ボーダーと影を同時に濃くしない
・暖色コーラルを面で使わない・彩度を上げてネオンにしない（danger/今すぐ の点のみ）
・見出しを 800 にしない（semibold --fw-sb 600 で締める）
・Toast で永続情報を伝えない（消えるため保存は Inline Alert）
12.2 フィードバックパターンの使い分け
パターン使いどころ避けるべき場面
Toast 完了通知 / 情報の瞬間的フィードバック（登録完了、削除完了） ユーザーが次に行動するための情報。エラー詳細
Inline Alert フォーム検証エラー、要確認（OCR）、進行情報。視野に残したい情報 一瞬で消えてよい完了通知
Loading Overlay 画面全体の非同期処理中（保存中、削除中、画面遷移時） 2秒以内に終わる軽い処理（ボタン内スピナーに）
Confirm Dialog 未保存の離脱、2段階確認。取り消し可能な操作 毎回の操作確認（操作を阻害）
Destructive Dialog Journey/Step 削除、アカウント削除。取り消し不能 可逆操作（Inline Alert で十分）

13. チケット風 UI パターン

Journey 内の Step を「疑似チケット」として視覚化し、最重要情報を一瞥で把握できるようにするための抽象化パターン。 過度な搭乗券の模倣は避け、teal hero + mono code に寄せた軽量な情報設計を主目的とする。対応実装候補: components/Ticket.tsx, app/trips/[id]/TripDetailClient.tsx

13.1 情報階層の三原則
PRIMARY · Hero
いつ・どこ

タップせず視認できるべき情報。時刻・空港コード・駅名・施設名など、行動の即時判断に必要なもの。font-size ≥ 22px, monospace 強調。

SECONDARY · Meta
行動に影響する属性

便名・ゲート・座席・号車・チェックアウト時刻など。12-15px のグリッドで 2-3 列に配置し、ラベル + 値で提示。

TERTIARY · Code
参照 / 認証

確認番号・予約番号・QR / バーコード。下段フッターゾーンに固定。monospace + letter-spacing で可読性。

14.1 目的と原則
