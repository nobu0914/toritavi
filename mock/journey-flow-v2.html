比較と推奨

観点 案 A：OCR 後に受け皿を聞く 案 B：先に Journey を作る 案 C：Journey 下書きに見せる
Journey 意識 中 — OCR 直後に「どこに入れる？」で意識が戻る 高 — 入口からタイトルが必要になる 高 — 結果が Journey 下書きとして現れる
既存ユーザー移行コスト 低 — 今の OCR 起点が残る 高 — 入口の既定動線が変わる 中 — 挙動が大きく変わる
速い追加の体験 軽い — 「新規」1 タップで作成 重い — タイトル入力の強制感 軽い — ただし推定に左右される
既存 Journey への追加 明示的 — 必ず分岐で選ぶ Journey 内から追加 — 直感的 自動マッチ — 精度次第でミス発生
未整理導線 3 分岐で常に選べる quick path に逃がす 低信頼時のフォールバック扱い
タイトル提案 新規作成時に chip で提案 テンプレあるが手打ち寄り 常に OCR ベースで chip 提案
実装スコープ 中 — 分岐画面 + タイトル提案を追加 大 — 入口フロー + /trips/new 強化 大 — 推論ロジック + UI 再構成
不確実性 低 — 既存のパーツで組める 低 — 伝統的な UI 高 — OCR 品質に依存する
RECOMMENDATION

案 A を採用

Toritavi の価値仮説は「チケットを撮れば旅程ができる」体験にある。 案 B は入口を変えすぎて「頻度が高い素早い追加」を重くする。 案 C は理想形に近いが OCR 品質に体験が人質にされる。 案 A は 最短で Journey 意識を取り戻し、既存導線を壊さず、未整理への逃げ道も自然に持てる。 案 C の "自動統合プレビュー" は将来 case A の上に増築する拡張として追える。

採用時の実装方針メモ（レビュー用）

  • 既存の ScanFlow.tsx の最後で「Step を保存して遷移」するパスを止め、登録先の選択画面（案 A-3）をブロッキングで表示する
  • 選択は "new" | "existing" | "unfiled" の 3 種。内部状態として OCR 結果は未保存で持ち、選択時に書き込みへ進む
  • "new" 分岐は /trips/new に OCR draft を流し込み、タイトル chip（地名 / 目的 / 日付 / 手入力）を デフォルト選択付き で表示する。ロジックは buildTitleSuggestions(ocr) を lib に新設
  • "existing" 分岐は 日付が近い Journey を 1 件ハイライト し、その他は通常一覧。Journey context card（タイトル / 日付 / step 数 / 次の予定 / mini timeline）をリストに出す
  • "unfiled" 分岐は既存の placeholder ページに実データを保持する仕組みを追加する必要あり（toritavi_unfiled_items テーブル？ 既存 Step を journey_id = NULL で持つ？ — 別途設計）
  • タイトル提案の候補生成: 出発・到着地 / 宿泊先 / 医療機関名 / 日付の組合せ。候補 3〜4 件 + "手入力"。最も信頼度が高いものを default selected
  • Journey 作成後の Step merge チェック（現行の StepMergeDialog）は A-4 以降で既存ロジックを流用
  • デザイン tokens は DS v2 のまま再利用。新規 UI（choice tile / title chip / context card / mini timeline）は既存 .card, .chip, .badge の派生で組めるので追加は最小

次フェーズ（MVP 以降 · 案 C から取り込む候補）

  • 複数便 / 往復 / PDF まとめ読み込みからの multi-step draft（C-3）
  • 同日既存 Journey の 自動合体プレビュー（C-2）— 現状の StepMergeDialog を Journey 粒度へ拡張
  • 未整理に貯めた item を一括で Journey 化する flow（unfiled → journey promotion）

スコープ外（今回やらない）

  • OCR 抽出精度の改善（モデル入替 / プロンプト変更）
  • Journey テンプレート（出張 / 通院 / 旅行 などのプリセット）
  • 共有・エクスポート周り
