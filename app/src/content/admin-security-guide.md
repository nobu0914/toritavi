## セキュリティ運用ガイド（管理者向け）

### 基本方針

JUNROS はユーザーが撮影したチケット・予約書類の画像（氏名・予約番号・住所等の個人情報を含む）を扱います。運用の最優先事項は **「ユーザーの写真を流出させない」** ことです。すべての判断はこの原則を基準にしてください。

### データ保護の仕組み（現行）

| 領域 | 仕組み |
|---|---|
| アクセス制御 | 全ユーザーデータは Supabase RLS で本人のみ閲覧可。画像バケット（`step-attachments`）は非公開＋所有者限定 |
| 通信 | 全通信 HTTPS（平文通信はアプリ設定レベルで遮断）|
| 端末内 | 撮影・選択画像の一時ファイルは読み取り後に即削除。ログアウト/退会時に全消去。OSバックアップへの混入も遮断 |
| 画像の付帯情報 | アップロード前に EXIF（GPS位置情報等）を必ず除去。除去できない画像は送信しない |
| 退会時 | サーバ側でデータベースと画像ストレージの双方を完全削除 |
| AI解析（OCR） | 画像は Anthropic API に送信され 30日保持ののち削除（プライバシーポリシー記載）|

### 定期チェック（頻度と内容）

| 頻度 | 内容 |
|---|---|
| リリース前 | 自動チェック一式（`bash tool/security_check.sh`）… テスト・流出防止の静的検査・本番アクセス制御の実地確認 |
| 週次 | クイック版（`--fast`）で静的検査のみ |
| 月次 | 本番RLS実地確認／依存ライブラリの脆弱性更新確認／Supabase Advisors の警告確認／RevenueCat Webhook 配信エラー確認 |
| 四半期 | 全域監査（コード全体のバグ・写真ライフサイクル追跡）／退会テスト（テストアカウントで画像が完全削除されるか）／プライバシーポリシーと実装の整合確認 |

### 月次の手動確認（ダッシュボード）

1. **Supabase → Table Editor**: `toritavi_journeys` / `toritavi_steps` / `toritavi_user_plan` の RLS が有効であること
2. **Supabase → Storage**: `step-attachments` が private であること
3. **Supabase → Advisors**: Security 警告がゼロであること
4. **RevenueCat**: Webhook 配信エラーがないこと（課金プラン同期の生命線）
5. **Supabase → Auth**: 不審なユーザー急増・異常なログイン失敗がないこと

### 異常時の初動

| 状況 | 初動 |
|---|---|
| キー露出の疑い | Supabase で anon key をローテーション＋全セッション失効 |
| アクセス制御の破れの疑い | `bash tool/rls_probe.sh` で実地確認 → 破れていれば該当テーブルの RLS を即時有効化 |
| 写真流出の疑い | Supabase Logs で Storage アクセスログを確認し影響範囲を特定 → 必要に応じ利用者へ通知 |

### 問い合わせ対応の要点

| 質問 | 回答 |
|---|---|
| 自分のデータは誰が見られる？ | 本人のみ。運営もダッシュボード経由でしかアクセスできず、アクセスは記録される |
| 退会したらデータは？ | アカウント・旅程・画像ともサーバから完全削除 |
| 画像はAIに送られる？ | OCR 時に Anthropic API へ送信、30日保持ののち削除。位置情報（EXIF）は送信前に除去済み |

### 定期チェックの実行（AI への指示文）

以下をそのまま Claude Code に貼って実行します。前提として「写真（スキャン画像）の流出防止が最重要観点。判断に迷ったら `docs/SECURITY_MAINTENANCE.md` の5原則に従う」。

**① フルチェック（リリース前 / 月次）**

```
toritavi_app のセキュリティ定期チェックを実行して。
1. bash tool/security_check.sh を実行（フル。テスト・静的不変条件・
   プラットフォーム設定・本番RLS実地プローブを含む）。
2. ❌/FAIL が出た項目は原因を調査し、修正案を提示（写真流出に関わるものは
   最優先・即修正）。⚠️ は内容を確認して報告のみでよい。
3. flutter pub outdated で supabase_flutter / purchases_flutter /
   firebase_* / image_picker / file_picker の更新有無を確認し、
   セキュリティ修正を含む更新があれば適用を提案。
4. 結果を「通過/要対応/手動確認が必要」の3区分でまとめて報告。
```

**② クイックチェック（週次 / コミット前）**

```
toritavi_app のクイックセキュリティチェックを実行して。
bash tool/security_check.sh --fast を実行し、FAIL があれば原因と修正案を報告。
全て通過ならその旨だけ簡潔に報告。
```

**③ 四半期・全域監査**

```
toritavi_app の四半期セキュリティ・品質監査を実行して。
1. bash tool/security_check.sh（フル）
2. 変更差分があれば /security-review
3. 全域バグスイープをワークフローで実行
   （機能領域ごとに finder → 敵対的検証で誤検出を落とす構成）
4. 写真ライフサイクル監査: 画像を扱うコードを列挙し、
   撮影→一時保存→アップロード→保存→表示→削除 を追跡。
   一時ファイルの削除漏れ・新規の外部送信先がないか確認。
5. docs/SECURITY_MAINTENANCE.md の手動チェックリストのうち
   コードから確認可能な項目を検証し、ダッシュボード確認が必要な
   項目は一覧にして提示。
```
